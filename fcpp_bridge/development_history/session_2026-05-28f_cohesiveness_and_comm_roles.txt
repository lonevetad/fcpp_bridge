Configuration (all configurable):
  NODES=15, SINK_POINTS=2, SOURCE_POINTS=3
  COMM=100.0, SIDE=213, NUM_ROUNDS=60, MSG_TICKS_INTERVAL=10
  ISOLATION_THRESHOLD=0.20, MAX_MIGRATION_ITERS=50, MAX_POINT_PLACEMENT_ITERS=10

Setup functions (pure Python, not aggregate):
  _build_and_migrate_network() — places nodes randomly, migrates isolated ones
  _place_points() — places sink/source points near anchor nodes (no conflicts)
  _assign_initial_roles() — Receiver: nearest to sink, Sender: nearest to source
  _setup_network() — orchestrates the above; raises ValueError if SINK+SOURCE > NODES

DSL aggregate function (7 steps):
  1. bis_distance(is_receiver) — gradient toward Receivers
  2. bis_distance(is_sender)   — gradient from Senders
  3. nbr + min_hood            — Sender election
  4. old(0, t+1)               — round counter
  5. broadcast(dist_from_sender, payload) — propagate Sender messages
  6. old({}, accumulate)       — received-message log at Receivers
  7. match/case                — per-role task + state assembly

Demo simulation output (15 nodes, 60 rounds, seed=42):
  2 Receivers, 3 Senders, 10 Repeaters
  12 distinct messages delivered to receivers (6 per receiver)
  3272 bytes of C++ generated

=== KEY DESIGN DECISIONS ===

1. neighbors_of signature: (positions, nid, comm) — takes positions dict as first arg
   so the function is pure (no closure over outer variables) and works for any
   deployment. Iterates positions.keys() instead of range(N) to avoid hardcoding
   the node count.

2. SPAWN_STATUS_* shared constants + local STATUS_* aliases:
   Aliases keep the existing examples' DSL code unchanged (STATUS_BORDER etc. still
   work), while the new example can use either form. No CALL-counter impact.

3. Role stability in communication_roles_assignment:
   Roles are set once in _assign_initial_roles() (pure Python, before simulation).
   The aggregate compute() reads role from self_state.role — never changes it.
   This matches the spec: "nodes won't change their CommunicationRole".
   The aggregate function nonetheless demonstrates the full role-election primitives
   (nbr + min_hood) even though they converge immediately to the pre-set roles.

4. message payload = (sender_uid, round_tick, dist_to_nearest_source):
   round_tick serves as the message counter; self_uid() (→ node.uid in C++) is the
   sender identifier. dist_to_nearest_source is dummy payload data showing the
   Sender's context. Future exercises can replace this with real data.

5. broadcast vs spawn for message routing:
   broadcast(dist_from_sender, payload) was chosen (not spawn) because:
   a) the spec suggests broadcast ("most likely, the broadcast");
   b) the new example already demonstrates spawn indirectly through worker_role_assignment;
   c) broadcast is simpler to explain for the single-channel scenario.
   spawn would be needed for FE-1 (multiple independent channels).
