Part 2 — Enum transpilation fix (v1.8.4)
-----------------------------------------
Problem discovered:
  The transpiler emitted dotted attribute chains verbatim.
  "case WorkerRole.RECEIVER.value:" in Python → "case WorkerRole.RECEIVER.value:"
  in C++, which is invalid because C++ case labels must be compile-time integer
  constants, not runtime attribute access expressions.

Root cause:
  PythonAstVisitor.visit_Attribute emitted "obj.attr" without any namespace
  context, so it couldn't resolve "WorkerRole.RECEIVER.value" to the integer 1.

Fix:
  1. transpiler/python_ast_visitor.py
     - PythonAstVisitor.__init__ now accepts optional "constants: dict = None"
     - New _resolve_dotted_chain(node) helper: walks the Attribute chain, looks
       up the root name in self.constants, resolves remaining attrs via getattr()
     - visit_Attribute: if constants is set, tries _resolve_dotted_chain; if it
       returns an int/float/bool, emits the literal value instead of the chain

  2. transpiler/transpiler_core.py
     - _transpile_method_body now passes getattr(method, '__globals__', {}) as
       "constants" to PythonAstVisitor so all module-level names (including
       IntEnum classes) are available for constant-folding

=== KEY DESIGN DECISIONS ===

1. enum folding only for int/float/bool
   IntEnum members satisfy isinstance(x, int) → fold to integer literal.
   bool maps to 1/0 (not true/false) to preserve integer type in C++.
   str, list, dict and other types are NOT folded → verbatim passthrough.

2. constants = method.__globals__ (full module namespace)
   Gives the visitor access to all module-level names, including IntEnum
   classes defined in the same file as the aggregate function. No explicit
   registration needed.

3. Backward-compatible default
   PythonAstVisitor() with no args → constants={} → no folding → all
   existing tests and code paths unaffected.

4. CALL-counter invariant still holds
   Enum folding only affects case labels and comparison RHS values, which
   are local-computation positions. No new primitives added; CALL counter
   alignment is unchanged.
