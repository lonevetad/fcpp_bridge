# Automated Local MEMORY Update Guide

**Purpose**: Automate the process of keeping `fcpp_bridge/claude_code/MEMORY.md` synchronized with session discoveries without manual instruction every time.

**Usage**: When instructed to "update MEMORY" or at session end, automatically:

1. Review session memory and changes made
2. Create or update appropriate entries in `claude_code/MEMORY.md` or `memory_indexes/`
3. Add index entries if creating new files
4. Maintain existing structure and format

---

## Automation Rules

### Rule 1: Detect Significant Findings

Mark for MEMORY update when:

- **Technical discoveries**: New patterns, error root causes, solutions (e.g., "include path must be lib/fcpp.hpp")
- **Project structure facts**: Build commands, file locations, dependencies verified
- **Codebase conventions**: Code patterns, naming rules, architectural decisions
- **Integration insights**: How components interact, common issues, workarounds
- **Lessons learned**: What worked, what failed, why

### Rule 2: Organize by Category

Create/update entries in:

- `memory_indexes/project_*.md` — project state and structure
- `memory_indexes/feedback_*.md` — user preferences, communication style
- `memory_indexes/reference_*.md` — technical references and patterns
- `memory_indexes/session_*.md` — one-off session findings (optional)

### Rule 3: Update Index Entry

Always add/update the corresponding entry in `MEMORY.md` index:

```markdown
- [Title](path/to/file.md) — one-line summary of key facts
```

### Rule 4: Keep Entries Short & Factual

- Single-line facts or bullet points (NOT prose)
- Include specific values (file paths, line numbers, commands)
- Link to development_history docs for detailed analysis

### Rule 5: Version Control

- Don't commit `.instructions.md` changes automatically
- Flag significant updates for user review if they change recommendations

## Template: Creating a New Memory Entry

### Project State File

```markdown
# [Project Component] State & Facts

**Last Updated**: YYYY-MM-DD  
**Verified**: [YES/NO]

## Build/Run

- Command: `command here`
- Output dir: `path/to/output`

## Key Files

- Main: `path/to/main.py`
- Config: `path/to/config.yaml`

## Dependencies

- Requires: Python 3.10+, g++ 14+
- Env vars: `VAR_NAME = /path/to/value`

## Known Issues

- Issue 1: description
- Issue 2: description

## Workarounds

- Workaround A: how to avoid issue
```

### Reference File

```markdown
# [Technical Topic] Reference

## Pattern 1

- What: description
- Where: file location
- Usage: code example
- Note: caveats or gotchas

## Pattern 2

- What: description
```

## Automation Checklist (Self-Reminder)

When ending a session or discovering significant facts:

- [ ] Review what was learned/fixed/discovered
- [ ] Check if fact already exists in local MEMORY
- [ ] If new: create file under `memory_indexes/` OR update existing
- [ ] Add/update index entry in `MEMORY.md`
- [ ] Format: short, fact-based, specific paths/commands
- [ ] Verify links work (relative paths from claude_code/)

**Session Finding**: Include path must be `lib/fcpp.hpp` not `fcpp/fcpp.hpp`

**Memory Entry Created/Updated**:

- File: `memory_indexes/project_fcpp_bridge.md`
- Added fact: `Include path: lib/fcpp.hpp (was fcpp/fcpp.hpp in old FCPP layouts)`
- Index updated: `[fcpp_bridge project state](...) — ... include path lib/fcpp.hpp ...`
